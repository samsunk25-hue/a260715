## 0. 가장 먼저 알아야 할 것 (이 문서의 핵심)

PRD를 뜯어보니 **Firebase가 필요한 기능은 딱 하나뿐**입니다.

PRD 2장에 "복잡한 개인정보나 별명 연동 없이"라고 못박혀 있습니다. 로그인이 없다는 뜻이고,
로그인이 없으면 **학생 개인 데이터는 애초에 서버에 올릴 이유가 없습니다**. 누구 것인지 구분할 방법이 없으니까요.
그래서 미션 체크, 누적 하트, 드래곤 레벨, 주간 일기는 전부 각자 기기에만 저장하면 됩니다.

반대로 "전체 응원"은 **교사와 학부모**가 쓴 걸 학생들이 봐야 하니 반드시 공유돼야 합니다.

즉 이렇게 갈라집니다:

| 데이터 | 저장 위치 | 이유 |
|---|---|---|
| 미션 체크 / 누적 하트 / 드래곤 / 주간 일기 | 브라우저 localStorage | 개인 것, 공유할 필요 없음 |
| **전체 응원 메시지** | **Firebase Firestore** | 교사·학부모 → 모든 학생, 공유 필수 |

**결과적으로 Firestore에는 컬렉션 하나, 날짜당 문서 하나, 필드 두 개만 들어갑니다.**
Firebase에서 배워야 할 건 "쓰기 1줄, 읽기 1줄"이 전부입니다. 2일 안에 충분합니다.

이 구조는 PRD를 타협한 게 아니라 PRD가 원래 의도한 구조입니다.
PRD 3장 ③번이 "당일 앱에 접속하는 **모든** 학생"이라고 했지 "학생별로"라고 하지 않았습니다.
전체 공지 하나만 공유하면 되는 겁니다.

## 2. 기술 스택

빌드 도구, npm, React 전부 **안 씁니다.** 초보자가 2일 안에 끝내려면 설정에 시간을 쓰면 안 됩니다.

| 항목 | 선택 | 이유 |
|---|---|---|
| 화면 | HTML + CSS + 순수 JavaScript | 배울 게 없음. 바로 시작 가능 |
| 개인 데이터 | localStorage | 브라우저 내장. 코드 2줄 |
| 공유 데이터 | Firebase Firestore (CDN) | 응원 메시지 전용 |
| 그래프 | Chart.js (CDN) | PRD가 직접 지정 |
| 꽃가루 | canvas-confetti (CDN) | 1줄로 끝남 |
| 실행 | VS Code + Live Server 확장 | ⚠️ 아래 주의사항 필독 |
| 배포 | Firebase Hosting | 이미 Firebase 프로젝트가 있으니 공짜로 딸려옴 |

### ⚠️ 초보자가 100% 걸리는 함정 — 미리 읽어두세요

**HTML 파일을 더블클릭해서 열면 Firebase가 작동하지 않습니다.**

Firebase는 `<script type="module">` 방식으로 불러오는데, 이건 `file://` 주소에서는
브라우저 보안 정책(CORS) 때문에 차단됩니다. 콘솔에 빨간 에러가 뜨고 아무것도 안 됩니다.

**해결책:** VS Code에서 `Live Server` 확장을 설치하고, `index.html`에서 우클릭 →
`Open with Live Server`. 주소가 `http://127.0.0.1:5500` 이런 식으로 뜨면 정상입니다.

이거 모르고 "Firebase가 고장났다"고 두 시간 날리는 사람이 정말 많습니다. **꼭 기억하세요.**

## 4. 데이터 설계

### localStorage (개인 데이터)

키 하나에 전부 담습니다. 키 이름: `studyDragon`

```js
{
  missions: {
    "2026-07-15": {
      tasks: [true, false, true],              // 선생님 과제 3개 체크 여부
      custom: { text: "영어 단어 20개", done: true }   // 자율 학습
    },
    "2026-07-14": { ... }
  },
  diaries: {
    "2026-07-12": "이번 주는 수학이 어려웠다..."   // 일요일 날짜만
  }
}
```

**누적 하트는 저장하지 않습니다.** `missions`를 전부 훑어서 `true` 개수를 세면 됩니다.
따로 저장하면 두 값이 어긋날 때 버그를 잡기 어려워집니다. 셀 때마다 계산하는 게 안전합니다.

### Firestore (공유 데이터)

교사와 학부모가 **각각** 응원을 쓰므로, 하루에 메시지가 여러 개 쌓입니다.
따라서 "날짜당 문서 하나"가 아니라 **"응원 하나당 문서 하나"**로 갑니다.

```
cheers  (컬렉션)
  ├─ {자동ID}
  │    ├─ date:      "2026-07-15"      ← 날짜를 필드로!
  │    ├─ message:   "오늘도 힘내자! 🔥"
  │    └─ createdAt:  타임스탬프
  ├─ {자동ID}
  │    ├─ date:      "2026-07-15"      ← 같은 날에 또 하나
  │    ├─ message:   "수학 수행평가 화이팅!"
  │    └─ createdAt:  타임스탬프
  └─ ...
```

- **쓰기:** `addDoc(collection(db, "cheers"), { date, message, createdAt: serverTimestamp() })`
  → `setDoc`이 아니라 **`addDoc`**입니다. `setDoc`은 덮어쓰고, `addDoc`은 새로 추가합니다. 이게 핵심 차이입니다.
- **읽기:** 달력을 열 때 **그 달 전체를 한 번에** 가져옵니다.
  ```js
  query(collection(db, "cheers"),
        where("date", ">=", "2026-07-01"),
        where("date", "<=", "2026-07-31"))
  ```

### 왜 날짜를 문서 ID가 아니라 필드로 두는가 (중요)

이렇게 하면 **한 번의 조회로 두 기능이 동시에 해결됩니다.**

1. 캘린더 **핑크 뱃지** (화면 1) — 응원이 있는 날짜가 어디인지
2. 미션 모달 **상단 응원 카드** (화면 2) — 그날 메시지 내용

만약 날짜를 문서 ID로 쓰면, 뱃지를 그리려고 **날짜마다 조회를 한 번씩** 해야 합니다.
한 달이면 31번입니다. 반면 날짜를 필드로 두면 위 쿼리 하나로 그 달 전체가 들어오고,
모달을 열 때는 **이미 받아둔 데이터에서 걸러 쓰기만** 하면 됩니다. 추가 조회가 0번입니다.

정렬(오래된 순)은 Firestore에 시키지 말고 **받아온 뒤 JS에서 `.sort()`** 하세요.
`where` + `orderBy`를 서로 다른 필드에 같이 쓰면 Firestore가 "복합 색인을 만들라"며 에러를 냅니다.
초보자가 여기서 잘 막힙니다. JS에서 정렬하면 그 문제 자체가 없습니다.

## 6. Firebase 설정 — Day 1의 관문

**Day 1 오전에 이걸 먼저 하는 이유:** Firebase가 막힐 거라면 20시간째가 아니라 2시간째에 알아야 합니다.
여기서 막히면 즉시 9장의 Plan B로 갈아탈 수 있고, 그래도 앱은 완성됩니다.

### 순서

1. https://console.firebase.google.com 접속 → **프로젝트 만들기** → 이름 `study-dragon`
   → Google 애널리틱스는 **꺼도 됩니다** (설정만 늘어남)
2. 프로젝트 안에서 **빌드 → Firestore Database → 데이터베이스 만들기**
   → **테스트 모드로 시작** 선택 → 위치는 `asia-northeast3 (서울)`
3. 좌측 상단 톱니바퀴 → **프로젝트 설정** → 아래로 스크롤 → **웹 앱 추가 (`</>` 아이콘)**
   → 앱 닉네임 아무거나 → **firebaseConfig 객체를 복사**
4. 복사한 걸 `firebase.js`에 붙여넣기
5. **스파이크 테스트:** 아무 버튼이나 만들어서 문서 하나 쓰고 읽어보기.
   Firebase 콘솔의 Firestore 화면에 `cheers` 컬렉션이 실제로 생기면 **성공**

### 보안 규칙 (2번의 "테스트 모드"는 30일 뒤 막힙니다)

Firestore → **규칙** 탭에 아래를 붙여넣고 게시하세요.

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /cheers/{id} {
      allow read: if true;

      allow create: if request.resource.data.message is string
                    && request.resource.data.message.size() > 0
                    && request.resource.data.message.size() < 200
                    && request.resource.data.date is string;

      // 응원은 쌓이기만 하고 사라지지 않는다
      allow update, delete: if false;
    }
  }
}
```

**마지막 줄이 중요합니다.** `allow update, delete: if false`는 한 번 올라간 응원을
**아무도 수정하거나 지울 수 없게** 데이터베이스 차원에서 막습니다.
교사가 학부모 메시지를 실수로 덮어쓰는 사고가 코드 버그와 무관하게 원천 차단됩니다.
"쌓기" 규칙을 앱 코드가 아니라 DB가 보장하게 만드는 겁니다.

**솔직하게 말씀드릴 점:** 이 규칙은 메시지 길이와 형식만 검사합니다.
로그인이 없으니 **주소를 아는 사람은 누구나 응원을 쓸 수 있습니다.**
교실 시연/과제 제출 수준에서는 괜찮지만, 진짜로 여러 학교에 배포할 거라면
나중에 Firebase Authentication을 붙여야 합니다. 지금은 범위 밖입니다.

그리고 지울 수 없게 막았으니, **부적절한 메시지가 올라가면 Firebase 콘솔에서 직접 지워야 합니다.**
(Firestore Database 화면에서 문서를 골라 삭제 — 콘솔은 이 규칙의 영향을 받지 않습니다.)
